# Conversational AI Vision

## 1. 문서 목적

이 문서는 Genshin Impact Story AI에서 최종적으로 만들고자 하는 **대화형 AI 경험**을 정의한다.

이 프로젝트의 최종 사용감은 ChatGPT처럼 자연스럽게 질문하고 답변받는 형태다. 하지만 현재 단계에서 바로 자유 대화형 AI를 구현하는 것이 목표는 아니다. 먼저 `basic_lookup`, `summary`, `analysis`, `research`, `guide` 같은 하부 라우트와 검색·근거 수집·검증 구조를 안정화하고, 그 위에 대화형 레이어를 올린다.

즉, 대화형 AI는 별도의 기능 하나가 아니라 다음 기능들을 묶는 최상위 오케스트레이션 레이어다.

```text
검색
정형 조회
요약
분석
연구
가이드
근거 고정
출처 표시
검증
대화 맥락 관리
```

---

## 2. 현재 답변이 템플릿처럼 느껴지는 이유

현재 시스템은 아직 ChatGPT식 대화형 답변 엔진이 아니라, 공식 데이터 기반 `basic_lookup`을 검증하는 단계다.

현재 구조는 대략 다음과 같다.

```text
사용자 질문
→ intent 분류
→ 공식 데이터에서 facts 추출
→ 템플릿 draft 생성
→ LLM rewriter
→ validator
→ 최종 답변
```

이 구조에서는 답변이 정해진 틀 안에서 생성되는 느낌이 강하다. 하지만 이는 실패가 아니라 의도된 초기 단계다. 공식 데이터 조회에서 환각을 줄이고, 정형 facts가 안정적으로 추출되는지 확인하기 위한 구조이기 때문이다.

최종 구조에서는 템플릿이 사라지는 것이 아니라 역할이 바뀐다.

```text
현재:
템플릿 = 최종 답변 형식

최종:
템플릿 = fallback draft / 사실 보존 기준 / validator 기준
```

최종 답변은 LLM Answer Writer가 자연스럽게 작성하되, 근거와 검증은 검색엔진과 Evidence Pack이 담당한다.

---

## 3. 최종 대화형 구조

최종 구조는 다음과 같다.

```text
사용자 질문
→ Conversation Orchestrator
→ Query Understanding
→ Query Frame 생성
→ Route Decision
→ Planner
→ Tool Execution
→ Evidence Pack 생성
→ Answer Writer
→ Validator
→ 최종 답변
```

각 계층의 역할은 다음과 같다.

| 계층 | 역할 |
|---|---|
| Conversation Orchestrator | 이전 대화 맥락, 대명사, 후속 질문 해석 |
| Query Understanding | 질문의 대상, 목적, 깊이, 필요한 정보 범위 파악 |
| Query Frame | 질문을 구조화한 내부 표현 |
| Route Decision | basic_lookup / summary / analysis / research / guide 중 선택 |
| Planner | 필요한 검색과 도구 호출 계획 생성 |
| Tool Execution | DB 조회, FTS 검색, Source Reader, Evidence Pack 생성 |
| Answer Writer | 근거를 바탕으로 자연어 답변 생성 |
| Validator | 사실 누락, 환각, 출처 수준, 해석/추측 혼합 여부 검증 |

---

## 4. Query Frame

ChatGPT처럼 “찰떡같이 알아듣는” 느낌은 단순 intent 분류만으로 만들 수 없다. 사용자의 질문을 여러 축으로 나누어 해석해야 한다.

예시 질문:

```text
푸리나는 스토리에서 어떤 캐릭터야?
```

내부 Query Frame:

```json
{
  "query": "푸리나는 스토리에서 어떤 캐릭터야?",
  "target": {
    "type": "character",
    "name": "푸리나"
  },
  "goal": "story_summary",
  "depth": "medium",
  "wanted_aspects": ["profile", "story_role", "character_arc", "related_events"],
  "source_policy": "official_first",
  "speculation_allowed": false,
  "needs_clarification": false
}
```

예시 질문:

```text
파네스와 천리가 같은 존재일 가능성은 있어?
```

내부 Query Frame:

```json
{
  "query": "파네스와 천리가 같은 존재일 가능성은 있어?",
  "target": {
    "type": "concept_relation",
    "names": ["파네스", "천리"]
  },
  "goal": "research",
  "depth": "deep",
  "wanted_aspects": ["direct_evidence", "indirect_evidence", "counter_evidence", "hypotheses"],
  "source_policy": "official_first",
  "speculation_allowed": true,
  "needs_clarification": false
}
```

Query Frame은 사용자의 말투와 질문 의도를 내부적으로 구조화하는 중간 표현이다. 이 구조가 있어야 후속 질문, 생략된 대상, 답변 깊이, 추측 허용 여부를 안정적으로 처리할 수 있다.

## 5. 라우트별 역할

### 5.1 basic_lookup

공식 데이터에 있는 정형 정보를 빠르고 정확하게 답하는 라우트다.

예시:

```text
푸리나 기본 정보
안개를 가르는 회광 효과
절연의 기치 4세트 효과
```

특징:

```text
DB와 규칙이 정답을 결정한다.
LLM은 말투만 다듬는다.
추측과 추천은 금지한다.
validator가 필수 필드 보존을 강하게 검사한다.
```

---

### 5.2 summary

특정 문서, 캐릭터 스토리, 개념, 사건을 요약하는 라우트다.

예시:

```text
푸리나 캐릭터 스토리 요약해줘
운명의 베틀이 뭐야?
폰타인 마신임무 핵심만 정리해줘
```

특징:

```text
검색 결과를 단순 나열하지 않는다.
Source Reader로 주변 문맥을 읽는다.
핵심 내용을 압축한다.
공식 근거에서 벗어나지 않는다.
```

---

### 5.3 analysis

여러 근거를 바탕으로 의미를 해석하는 라우트다.

예시:

```text
세계수 기억 조작이 스토리에 주는 의미가 뭐야?
푸리나는 왜 폰타인 스토리에서 중요한 인물이야?
셀레스티아와 천리의 관계는 어떻게 봐야 해?
```

특징:

```text
공식 사실과 해석을 분리한다.
근거를 바탕으로 의미를 설명한다.
반례나 주의점을 함께 제시한다.
너무 과감한 추측은 research로 넘긴다.
```

---

### 5.4 research

복수 개념을 깊게 조사하고, 가능한 가설을 세우며, 근거와 반례를 함께 정리하는 라우트다.

예시:

```text
파네스와 천리가 같은 존재일 가능성을 조사해줘
강림자와 세계수 기억 조작의 관계를 깊게 분석해줘
원신 결말 추측에서 천리, 셀레스티아, 파네스, 네 그림자를 연결해서 조사해줘
```

research는 이 프로젝트에서 가장 창의적인 라우트다.

중요한 점은 research가 반드시 안전하고 보수적인 결론만 내야 하는 것은 아니라는 것이다. 원신 스토리 추측은 본질적으로 불확실한 단서들을 연결하는 작업이므로, research 라우트에서는 과감하고 창의적인 가설을 생성할 수 있어야 한다.

다만 조건이 있다.

```text
가설은 가설로 표시한다.
공식 사실처럼 말하지 않는다.
어떤 근거에서 나온 추론인지 밝힌다.
반례와 약점을 함께 제시한다.
신뢰도 또는 추측 강도를 표시한다.
근거가 거의 없으면 “상상에 가까운 가설”로 분리한다.
```

즉 research 라우트는 “헛소리를 금지하는 라우트”가 아니라, **헛소리처럼 보일 정도로 과감한 가설도 근거·반례·불확실성 표시와 함께 다룰 수 있는 라우트**다.

research의 핵심은 정답을 단정하는 것이 아니라, 가능한 해석 공간을 넓히고 그 안에서 근거의 강도를 비교하는 것이다.

---

### 5.5 guide

캐릭터 사용법, 운용, 빌드, 파티 조합 등을 다루는 라우트다.

예시:

```text
푸리나 어떻게 써?
라이덴은 어떤 역할이야?
이 무기 누구한테 좋아?
```

guide는 공식 설정 답변과 다르다. 공식 데이터만으로는 실제 메타 추천을 완전히 확정할 수 없기 때문이다.

따라서 guide는 source policy를 분리해야 한다.

```text
L0: 공식 데이터
L1: 공식 데이터에서 계산 가능한 정보
L2: 자체 규칙 기반 해석
L3: 위키/공략 사이트
L4: 커뮤니티 메타
L5: AI 추론
```

초기 guide는 공식 데이터와 자체 해석 중심으로 제한하고, 외부 공략 데이터가 들어오면 출처 레벨을 명확히 표시한다.

## 6. Research 라우트의 창의성 정책

research 라우트는 다른 라우트보다 LLM의 창의성을 더 많이 허용한다.

단, 창의성은 사실 생성이 아니라 **가설 생성**에 사용되어야 한다.

### 6.1 허용되는 것

```text
멀리 떨어진 개념 사이의 연결 가능성 제안
상징, 모티프, 번역 차이 기반 해석
여러 가설 병렬 제시
가능성은 낮지만 흥미로운 추측 제시
반례가 있는 가설도 조건부로 다루기
```

### 6.2 금지되는 것

```text
근거 없는 내용을 공식 설정처럼 말하기
확정되지 않은 가설을 사실처럼 단정하기
출처가 없는 고유명사나 설정 만들기
반례를 의도적으로 숨기기
낮은 근거의 가설을 높은 신뢰도로 포장하기
```

### 6.3 research 답변의 권장 구조

```text
1. 짧은 결론
2. 확정 가능한 공식 사실
3. 가능한 가설 목록
4. 각 가설의 근거
5. 각 가설의 약점과 반례
6. 창의적이지만 근거가 약한 확장 가설
7. 현재로서는 확정할 수 없는 부분
8. 추가로 확인해야 할 문서
```

### 6.4 가설 강도 라벨

research 답변은 가설의 강도를 표시해야 한다.

```text
강함: 직접 근거가 여러 개 있음
중간: 간접 근거가 있고 반례가 치명적이지 않음
약함: 모티프나 정황 근거 중심
상상적: 흥미롭지만 공식 근거는 부족함
```

예시:

```text
가설 A — 중간
파네스와 천리가 같은 존재일 가능성

근거:
- ...
- ...

약점:
- ...

판단:
가능성은 있지만 현재 공식 텍스트만으로는 확정할 수 없다.
```

## 7. Answer Writer의 역할

최종 대화형 AI에서 Answer Writer는 템플릿을 그대로 출력하는 존재가 아니다.

Answer Writer는 다음 입력을 받는다.

```text
사용자 질문
Query Frame
Route
Evidence Pack
Source Policy
Answer Style
Validator Rules
```

그리고 질문에 맞춰 답변 구조를 자유롭게 만든다.

하지만 자유도는 표현과 구성에만 허용된다. 사실 주장은 Evidence Pack에 묶여야 한다.

```text
허용:
- 자연스러운 설명
- 사용자의 질문 깊이에 맞는 답변 길이
- 핵심부터 말하기
- 가설을 여러 개로 나누어 설명
- 창의적 해석 제안

금지:
- 근거 없는 사실 추가
- 출처 없는 확정 표현
- 공식 사실과 추측 혼합
- 필수 정보 누락
```

## 8. Validator의 역할

Validator는 단순 금지어 필터가 아니다.

최종 Validator는 route별로 달라져야 한다.

### basic_lookup validator

```text
필수 facts가 누락되었는가?
공식 데이터에 없는 숫자나 효과가 추가되었는가?
추천, 티어, 추측이 섞였는가?
```

### summary validator

```text
요약이 원문 범위를 벗어났는가?
핵심 사건이나 개념이 누락되었는가?
원문에 없는 인과관계를 만들었는가?
```

### analysis validator

```text
공식 사실과 해석이 분리되었는가?
근거 없는 단정이 있는가?
반례나 주의점이 누락되었는가?
```

### research validator

```text
가설이 가설로 표시되었는가?
근거와 반례가 함께 제시되었는가?
가설 강도가 표시되었는가?
공식 사실처럼 말한 추측이 있는가?
```

research validator는 창의적 가설 자체를 막지 않는다. 대신 그 가설이 어떤 근거 수준에 있는지 표시하도록 강제한다.

## 9. 대화형 UX의 목표

최종 사용자는 시스템이 다음을 해주길 기대한다.

```text
대상을 알아서 파악한다.
후속 질문을 이어받는다.
답변 깊이를 조절한다.
공식 정보와 추측을 구분한다.
스토리 질문과 성능 질문을 다르게 처리한다.
필요하면 근거와 반례를 보여준다.
출처를 클릭해 원문을 볼 수 있다.
```

예시 대화:

```text
사용자: 푸리나 기본 정보 알려줘
AI: 푸리나의 기본 프로필과 짧은 설명을 제공

사용자: 스토리에서는?
AI 내부 해석:
- "스토리에서는?" = 푸리나의 스토리상 역할 질문
- route = summary 또는 analysis
- target = 이전 턴의 푸리나

사용자: 그럼 천리랑 연결될 가능성도 있어?
AI 내부 해석:
- target = 푸리나 + 천리
- route = research
- speculation_allowed = true
```

이 흐름이 가능해지면 사용자는 “ChatGPT처럼 알아듣는다”고 느끼게 된다.

## 10. 구현 순서

대화형 AI는 최종 목표지만, 구현 순서는 다음이 맞다.

```text
1. basic_lookup 안정화
2. facts extractor 확장
3. route별 validator 강화
4. Source Reader 구현
5. summary 라우트 구현
6. analysis 라우트 구현
7. research 라우트 구현
8. guide 라우트 구현
9. Query Frame 도입
10. Planner 도입
11. Conversation Orchestrator 구현
12. API / 웹 UI 구현
```

즉 지금 당장 웹 채팅 UI나 자유 대화형 답변을 먼저 만들면 안 된다. 먼저 하부 라우트가 제대로 동작해야 한다.

## 11. 최종 정리

이 프로젝트가 지향하는 대화형 AI는 단순히 LLM에게 질문을 던지고 자연스럽게 답하게 만드는 챗봇이 아니다.

최종 목표는 다음과 같다.

```text
사용자가 자연어로 질문하면
시스템이 질문 의도와 맥락을 이해하고
필요한 라우트를 선택하고
공식 데이터와 스토리 문서를 검색하고
근거와 반례를 고정하고
LLM이 자연스럽게 답변하며
Validator가 사실과 추측의 경계를 검사하는
대화형 원신 스토리 연구 AI
```

research 라우트에서는 특히 창의적인 가설 생성을 허용한다. 단, 그 가설은 공식 사실처럼 포장되어서는 안 되며, 반드시 근거·반례·가설 강도와 함께 제시되어야 한다.

따라서 이 프로젝트의 대화형 AI 방향은 다음 한 문장으로 정리할 수 있다.

```text
ChatGPT처럼 자연스럽게 대화하되, 공식 사실·해석·추측·창의적 가설의 경계를 끝까지 유지하는 원신 스토리 연구 AI.
```
